# 🚀 استراتيجية نشر وتحديث النظام (Deployment Strategy)

## الهيكلية (Architecture)
نستخدم معمارية **Multi-tenant (SaaS)**، مما يعني أن جميع العملاء يتشاركون في قاعدة بيانات واحدة مع عزل منطقي تام للبيانات.

### لماذا؟
1. **سهولة الإدارة:** تحديث واحد لقاعدة البيانات يشمل جميع العملاء فوراً.
2. **كفاءة الموارد:** تقليل التكلفة عبر استخدام مشروع Supabase واحد لخدمة عدة شركات.
3. **التحكم المركزي:** إمكانية إدارة الاشتراكات والصلاحيات لجميع المنظمات من لوحة تحكم واحدة.

---

## كيف يعمل النظام مع تعدد الشركات؟

### 1. الكود (Frontend)
*   لديك **مستودع Git واحد** (Repository) يحتوي على كود React.
*   عندما تريد تحديث البرنامج لجميع العملاء:
    1.  قم بتعديل الكود محلياً.
    2.  ارفع التعديلات (`git push`).
    3.  يتم تحديث الواجهة لجميع العملاء في نفس الوقت، حيث يتم التعرف على هوية الشركة عبر تسجيل الدخول.

### 2. قاعدة البيانات (Backend)
*   يتم استخدام مشروع Supabase واحد وقاعدة بيانات واحدة لجميع الشركات.
*   **عند انضمام عميل جديد:**
    1.  يتم إنشاء سجل جديد في جدول `organizations`.
    2.  تقوم دالة `initialize_egyptian_coa` بتأسيس دليل الحسابات الخاص بالشركة الجديدة آلياً.

*   **عند تحديث قاعدة البيانات (مثلاً إضافة جدول جديد):**
    *   يتم تنفيذ التحديث مرة واحدة فقط، وسيستفيد منه جميع العملاء بفضل وجود `organization_id` في كل الجداول.
